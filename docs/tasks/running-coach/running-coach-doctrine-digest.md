## Canonical Non-Negotiables

- No fake precise pace.
- No fake personal HR.
- Target time alone is not pace truth.
- Executable HR targets require personal HR-zone truth.
- Age-estimated/default HR ranges are advisory/readback only.
- Missing pace/HR truth must fall back to structure-only executable anatomy or effort/cue language,
  not invented biometric precision.
- Backend validation and deterministic plan-building own training truth.
- Frontend renders backend-shaped workout identity, metric mode, endpoint, and safety copy.
- AI may draft or enrich only inside backend validation and review boundaries.
- Risky plan creation, replacement, refresh, or application remains reviewed/confirmed.

## Family Promises And Non-Promises

| Family | Promise | Non-promise |
| --- | --- | --- |
| `10K` | Visible 10K repeatability, rhythm, and exact `10000m` endpoint when selected-distance completion is accepted | No fake threshold/advanced specificity for unsupported runners |
| `Half Marathon` | Visible 21.1K durability, long-run development, and exact `21100m` endpoint | No generic tempo-only plan when half-specific durability is required |
| `Marathon Base` | Honest marathon durability/base-building with base endpoint language | Not full marathon race readiness and not exact `42195m` selected-distance completion |
| `Marathon Completion` | Separate completion-specific selected-distance family with exact `42195m` endpoint when implemented/accepted | Not target-time/race-pace fantasy and not a rename of Marathon Base |
| Ultra / mountain / trail | Time-on-feet, terrain, climbing/descending, hike-run, and durability context where supported | No exact elevation prescriptions or route-matching promises |

## Eligibility, Auto-Extension, And Valid Blocks

Hito should avoid dead ends for coach-plausible runners. When a selected-distance goal is unsafe or
not credible inside the requested horizon, the preferred path is conservative auto-extension or a
clear reviewed unavailable state, not silent collapse into generic filler.

Valid blocks include:

- impossible rest-day / long-run-day constraints
- too few running days for the requested family and horizon
- stale or unsupported setup input
- validation failure that would otherwise create unsafe load, fake metric truth, or endpoint dishonesty

Invalid blocks include:

- refusing simply because the runner is beginner-level when a bridge/extension can produce a safe plan
- refusing because conservative load removes advanced specificity, if family identity can still be
  preserved safely
- hiding endpoint or richness failure behind a technically valid generic plan

## Workout Grammar And Richness Bar

Runner-facing richness must survive preview, confirm, saved calendar readback, and export. A plan is
not acceptable just because internal composition metadata looks varied.

Minimum rules:

- family truth must be visible in workout identities and row titles
- long-run richness alone is insufficient
- non-long-run identity must recur enough to avoid long identity deserts
- the middle and late plan cannot collapse into repeated easy/long/recovery filler
- long-run detail must evolve through checkpoints, finish cues, terrain/durability context, or
  endpoint-specific purpose where appropriate
- taper and cutback weeks must reduce load without erasing the family promise
- accepted preview richness must survive canonical review/export shaping

## Metric Truth

Pace:

- Only trusted benchmark truth, currently direct custom recent-5K support where accepted, may create
  broad runner-level pace ranges.
- Target time without benchmark support remains goal context, not pace truth.
- No-watch or unsupported execution contexts should not emit precise watch-style pace targets.

Heart rate:

- Personal HR targets require personal HR-zone truth.
- Default/age-estimated HR can appear as advisory readback, not executable target truth.
- Future HR-zone/AeT work must confirm runner-level truth before affecting plans.

Structure-only execution:

- When pace/HR truth is absent, non-rest workouts can still be executable through duration,
  distance, repeats, recoveries, segment anatomy, RPE, and clear purpose cues.

## Compression Rules

Safe to summarize after this digest exists:

- repeated audit narration
- long scenario inventories already represented by final verdicts and evidence links
- stale intermediate findings superseded by later accepted repair
- duplicate context copied across several audit reports

Do not summarize away:

- active family promises
- exact endpoint distances
- no fake pace / no fake personal HR
- no-dead-end / auto-extension rules
- richness gates and export/readback parity
- validation evidence references
- safety concerns and conservative-load boundaries

First safe D26 candidate:

- guarded compression of historical audit/acceptance reports only, not active doctrine contracts
- keep the batch around files already classified as historical evidence
- stop if a file contains unique doctrine not represented in this digest or current docs
